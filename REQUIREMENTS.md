# Requirements Specification: uCentral Client Daemon

This document lists the strict, numbered requirements for the Go-based uCentral Client daemon (`TIP-olg-ucentral-client`). All architectural designs, specifications, code implementations, and test suites must map directly back to these requirements.

---

## 1. Network Connectivity & Lifecycles

*   **REQ-001 (Concurrent Startup Loops):** The daemon must launch separate, independent, concurrent connection loops to NATS and the Cloud WebSocket at boot. A failure or delay in NATS connection must not block the Cloud connection, and vice versa.
*   **REQ-002 (Decoupled Connection State Machine):** The daemon must manage the Cloud and NATS connection lifecycles entirely independently. Their individual connection states (`Connecting` $\leftrightarrow$ `Connected`) must be evaluated continuously to form a composite Derived Status consisting of five distinct, globally observable connection states:
    *(Note: Capitalized names below are used for architectural documentation; the actual wire/log values strictly serialize as lowercase snake_case, e.g., `operational`, `cloud_degraded`).*
    *   `Operational`: Cloud is `Connected`, NATS is `Connected`.
    *   `CloudDegraded`: Cloud is `Connecting`, but NATS is `Connected`. (Daemon safely buffers telemetry locally; reconnects to Cloud with randomized exponential backoff of 2s-300s).
    *   `NATSDegraded`: NATS is `Connecting`, but Cloud is `Connected`. (Daemon fast-fails incoming Cloud commands with `local_service_unavailable`).
    *   `Connecting`: Both Cloud and NATS are `Connecting` (neutral startup/reconnect status).
    *   `ProtocolFailure`: Cloud is connected but protocol verification failed (e.g., Cloud explicitly rejects the client version). This state takes strict precedence over `NATSDegraded`; if verification fails, the state is `ProtocolFailure` regardless of NATS connection status.
    *   No daemon restart is allowed for connection recovery on either network layer.
*   **REQ-003 (Version Verification Fallback):** The daemon must transmit its supported protocol versions in the free-form `connect.capabilities` JSON payload. Since the OWGW protocol does not define a formal negotiation response, the daemon assumes protocol verification success if the WebSocket `connect` succeeds (error=0). The composite derived status must still be evaluated from both link states; therefore, it becomes `Operational` only when NATS is also connected, otherwise it remains `NATSDegraded`. If a future OWGW extension explicitly rejects the version via a fatal error response to `connect`, the daemon must enter `ProtocolFailure`. In this state, it remains connected for health reporting only and rejects all other commands with `local_service_unavailable` (JSON-RPC code -32603, application_code 3). The protocol verification state must be reset whenever the Cloud connection closes, ensuring a new WebSocket session performs verification again.

## 2. NATS & JetStream Schema

*   **REQ-004 (Subject Schema Versioning):** All NATS subjects used by the client must be versioned with a `v1` prefix and follow target-serial isolation boundaries:
    *   `ucentral.v1.device.<own-serial>.config.apply` (Request-Reply)
    *   `ucentral.v1.device.<own-serial>.action.<command>` (Request-Reply)
    *   `ucentral.v1.device.<own-serial>.state` (Pub-Sub)
    *   `ucentral.v1.device.<own-serial>.telemetry` (Pub-Sub)
    *   `ucentral.v1.device.<own-serial>.log` (Pub-Sub)
    *   `ucentral.v1.device.<own-serial>.health` (Pub-Sub)
    *   `ucentral.v1.device.<own-serial>.capabilities.get` (Request-Reply)
    *   `ucentral.v1.device.<own-serial>.status.get` (Request-Reply, downstream device/local-agent status query)
*   **REQ-005 (Permissive Parameter Validation):** The client must validate incoming configuration schemas using a compiled-in schema validator. It must enforce **permissive validation**: known schema parameters are strictly validated, but unknown future parameters must be passed through to NATS to preserve forward compatibility.
*   **REQ-006 (JetStream KV Consistency Contract):** The client must write configurations to JetStream KV (`cfg_desired` bucket) under key `desired.<serial>`. The NATS configure trigger must carry the target `uuid`, `kv_key`, and the NATS `kv_revision` to allow the downstream agent to fetch the configuration and verify ordering. Downstream agents must abort if the KV revision is higher than the trigger revision, and must not apply without a matching trigger revision.

## 3. Transaction & Request Management

*   **REQ-007 (Transaction Lifecycle):** Every incoming Cloud command must be tracked by an active in-memory transaction transitioning through: `Created` $\rightarrow$ `PreparingDispatch` $\rightarrow$ `PendingPublish` $\rightarrow$ `InFlight` $\rightarrow$ `Completed` / `Failed` / `TimedOut`. Any local setup failure (e.g. KV write error, payload error) or network publish failure (e.g. dispatch buffer full, NATS unavailable) must transition the transaction to `Failed` and perform cleanup, never to `TimedOut`. Transactions in `PreparingDispatch` or `PendingPublish` must be governed by a bounded dispatch deadline. If the deadline expires before successful NATS publication or request transmission, the transaction must transition to `Failed`, perform normal terminal cleanup, release any held state-changing reservation, and return `local_service_unavailable` (JSON-RPC -32603, application_code 3). It must not transition to `TimedOut`, because downstream execution was never confirmed.
*   **REQ-008 (Concurrency Serialization):** The Request Manager must serialize state-changing and security-sensitive commands (`configure`, `reboot`, `factory`, `upgrade`, `certupdate`, `reenroll`, `script`, `leds`) for the device. The Request Manager must atomically acquire/reserve the state-changing lock during transaction creation before the transaction becomes visible in `Created`. If a serialized transaction is active (`Created`, `PreparingDispatch`, `PendingPublish`, or `InFlight`), new serialized commands must be immediately rejected with a `busy` status (Error Code -32603). Read-only downstream queries and diagnostics (`capabilities.get`, `status.get`, `ping`, `trace`, `telemetry`, `remote_access`) must run in parallel and must not acquire the device state lock. The `status.get` subject is owned by the downstream device/local agent; the uCentral client must only publish requests to it and must not respond on it.
*   **REQ-009 (Request Lifecycle and Duplicate Rejection):** Before creating a transaction, the Request Manager must canonicalize the incoming Cloud JSON-RPC ID, including the Cloud Session ID, and perform the following checks in order: (1) If the canonical Cloud ID exists in the completed `TransactionCache` for the *same active session* and the entry has not expired, replay the cached response directly to the Cloud. This path must not create a new transaction, generate a new `rpc_id`, acquire the state-changing lock, write to NATS, or update `activeCloudRequests`. (2) If the canonical Cloud ID matches an active transaction in `Created`, `PreparingDispatch`, `PendingPublish`, or `InFlight`, reject the request with JSON-RPC `-32603` (busy). (3) Only if neither cache replay nor active duplicate rejection applies may the Request Manager create a new transaction and enter `Created`. The cache lookup, active-map lookup, active-map reservation, deterministic rpc_id formatting, and transaction creation must be performed atomically under one Request Manager synchronization boundary.
*   **REQ-010 (Operation-Specific Caching & TTL):** The transaction cache must persist results in-memory with TTLs categorized by command type to support duplicate request replay. Strict session isolation must be enforced: cached responses from an old WebSocket session are *never* replayed to a new session merely by reusing the same JSON-RPC ID. Recovery across reconnects requires the Cloud to use an explicit status query or `operation_id` mechanism. These TTLs must be configurable via environment variables, defaulting to the following if unset:
    *   `configure`, `leds`: Configurable via `OLG_CACHE_TTL_CONFIGURE` and `OLG_CACHE_TTL_LEDS` (default: 5 minutes)
    *   `reboot`, `remote_access`: Configurable via `OLG_CACHE_TTL_REBOOT` and `OLG_CACHE_TTL_REMOTE_ACCESS` (default: 10 minutes)
    *   `factory`, `certupdate`, `reenroll`, `script`: Configurable via `OLG_CACHE_TTL_FACTORY`, `OLG_CACHE_TTL_CERTUPDATE`, `OLG_CACHE_TTL_REENROLL`, and `OLG_CACHE_TTL_SCRIPT` (default: 30 minutes)
    *   `upgrade` (Firmware): Configurable via `OLG_CACHE_TTL_UPGRADE` (default: 60 minutes from completion)
    *   Read-only diagnostics (`ping`, `trace`, `telemetry`, `capabilities.get`, `status.get`): Configurable via `OLG_CACHE_TTL_DEFAULT` (default: 2 minutes)
*   **REQ-011 (Asynchronous Upgrade Tracking & Crash Recovery):** Firmware upgrades must run as a background operation tracked by a persistent `operation_id` independent of the initial JSON-RPC `id`.
    *   **Generation and Storage:** The uCentral client MUST generate a globally unique `operation_id` (e.g., a UUID), store the upgrade record durably (in a separate `OperationStore`), and acquire the state-changing lock *before* dispatching the long-running upgrade command via NATS. Upgrade operation records remain in `OperationStore` while the operation is active and are marked terminal or removed after terminal processing is durably completed.
    *   **Immediate Response:** The immediate standard OWGW response (0: Accepted/started, 1: Firmware invalid, 2: Required signature absent) is the *only* JSON-RPC response for the upgrade request. The daemon must send this response using the original JSON-RPC `id` while retaining the state-changing lock for the background operation. The daemon must *never* send a second ordinary JSON-RPC response with the same ID upon completion.
    *   **Agent Preservation:** The downstream upgrade agent MUST persist the received `operation_id` with its upgrade state before starting the upgrade. Every status response and upgrade result MUST return the same `operation_id`. If `active=true` or a long-running status is returned, the agent MUST include the `operation_id`.
    *   **Restart Recovery:** On startup, the uCentral client must load the active operation from the `OperationStore`, restore the state-changing lock, call downstream `status.get`, and correlate the response using `operation_id`. Each recovery `status.get` query MUST generate a fresh internal `rpc_id`; the persisted `operation_id` remains the durable identity used to associate the status response with the original upgrade operation. If the agent reports `active=true` but omits the `operation_id`, the client MUST treat it as an indeterminate recovery error, retain the state lock to prevent new commands, and MUST NOT generate a replacement ID. Additionally, the daemon must recover any terminal (`Active=false`) records pending Cloud delivery, enqueue their optional notifications, and then delete them.
    *   **Cloud Notifications & Status:** Terminal state is retained internally. Later terminal updates are either optional negotiated `upgrade_progress` notifications, or exposed after reconnect through an explicitly defined status/recovery extension. Sending `upgrade_progress` notifications is an optional extension and must only be sent when the connected gateway explicitly advertises support for that extension.

## 4. Queues & Outbound Traffic Scheduling

*   **REQ-012 (Command Dispatch Buffer):** The client must use a configurable, short-lived NATS dispatch buffer (default size: 100). If NATS is down or the buffer is full, incoming commands must fail fast with `local_service_unavailable` (JSON-RPC code -32603, application_code 3).
*   **REQ-013 (Command Result Priority Queue):** NATS command execution results must be processed through a bounded result queue (default size: 50). Because state-changing commands are serialized per device, this queue is not expected to fill during normal operation. The queue must never block core network or NATS subscriber loops. If it nears capacity, telemetry/log forwarding must be throttled. If the queue overflows, the daemon must record `command_result_overflow` and log the `rpc_id`, command type, and subject. If the result is correlated to an active transaction, the exact downstream result must be passed directly to the Request Manager and processed according to the true downstream outcome. The daemon must not replace the downstream result with a generated error, must not rewrite a successful transaction as Failed, and must not wait for the transaction timeout. The exact final Cloud response must be preserved in the `TransactionCache`.
*   **REQ-014 (WebSocket Outbound Priority Scheduler):** Outbound WebSocket traffic must be written via a priority scheduler:
    *   `Priority 0 (Highest)`: JSON-RPC responses. It bypasses lower-priority backlog but uses a dedicated bounded emergency queue. If exhausted, the daemon must preserve the terminal transaction state and cached response, treat the WebSocket writer path as unhealthy, trigger path recovery, and record an overflow metric.
    *   `Priority 1`: Audits, system crash logs, and health snapshots. To prevent blocking core NATS handler execution, `Push()` operations to Priority 1 must be non-blocking. If the queue reaches capacity, it must return a fast error and record an `audit_delivery_failure` metric instead of blocking the caller.
    *   `Priority 2`: Coalesced state metrics. If the scheduler queue is full, `Push()` must fail-fast. The producer must handle this by doing nothing; the state remains safely in the upstream `StateCoalescer` (via `Peek()`) so it can be re-attempted on the next tick with any newer state natively applied.
    *   `Priority 3 (Lowest)`: Telemetry events and standard logs. If the scheduler queue is full, `Push()` must fail-fast. The producer must drop the popped telemetry event and record a `dropped_by_reason.scheduler_full` metric, relying on the upstream `TelemetryRingBuffer` to accumulate new data.
    *   **Anti-Starvation Rule:** To prevent a flood of Priority 0 traffic from permanently starving telemetry and health, the scheduler must enforce a consecutive yield limit. After a maximum of 10 consecutive Priority 0 messages are yielded, the scheduler must forcefully yield at least one available message from the next highest populated queue (Priority 1, 2, or 3) before returning to Priority 0.
*   **REQ-015 (State Coalescer & Telemetry Ring Buffer):** 
    *   State statistics must be rate-limited to 1 message per 10s using a last-write-wins coalescer (newer reports overwrite older un-sent reports).
    *   Telemetry events must be rate-limited to 50/sec. On buffer overflow, the oldest events must be dropped (FIFO drop).
    *   The client must track drop counters via `dropped_by_reason` metrics.

## 5. Security & Observability

*   **REQ-016 (NATS Security & Target Isolation):** The daemon must connect using NKeys or JWT credentials and restrict its publish/subscribe permissions to subjects containing its `<own-serial>` only, and must explicitly include subscribe permissions for reply-inboxes (`_INBOX.>`) to support NATS request-reply flows.
*   **REQ-017 (Local Management Signal Security):** The local capability refresh trigger must be exposed as a Unix domain socket. Access must be restricted to root-only file permissions, and must be rate-limited and audit logged.
*   **REQ-018 (Audit Logging & Loop Prevention):** Every sensitive action (`reboot`, `factory`, `upgrade`, `certupdate`, `reenroll`, `script`) must generate a high-severity audit log forwarded to the Cloud. The audit record must include metadata (method, initiator, target, outcome) but never sensitive payloads (private keys, script source, signatures, tokens, full output). If forwarding fails, the client must increment `audit_delivery_failure` but must not generate another log, preventing recursive logging loops.
*   **REQ-019 (NATS-Native Health Reporting):** To maintain security and efficiency, the client must not expose HTTP ports. It must subscribe to device health snapshots on `ucentral.v1.device.<own-serial>.health` for Cloud forwarding. The uCentral client must not serve daemon liveness/readiness as a NATS responder on `ucentral.v1.device.<own-serial>.status.get`; that subject is reserved for downstream device/local-agent status queries. The daemon's own liveness, readiness, Cloud connectivity, NATS connectivity, queue depth, uptime, and local metrics must be tracked internally and reported to the Cloud through the WebSocket control path.

## 6. Sizing & Error Mapping

*   **REQ-020 (Sizing Constraints & Memory Protection):** Maximum uncompressed payload sizes must be strictly enforced: Configuration (10MB), State (1MB), Telemetry (256KB), Logs (64KB), Script Source (1MB), Certificate Bundle (2MB), and Script Output/Result (10MB). 
    *   **Zip-Bomb Protection:** For compressed payloads, limits must apply to the decompressed byte stream, and decompression must stop once the applicable uncompressed limit is exceeded.
    *   **Buffer Protection:** A hard frame-size limit must also be enforced at the raw transport layer (e.g. 11MB). This is required because underlying WebSocket libraries (like `gorilla/websocket`) buffer incoming network frames in memory; without a transport limit, a malicious uncompressible 1GB frame would OOM the device before decompression even starts.
    *   **Validation:** Inbound and outbound compression handling must be treated separately; outbound compression must not allow payloads to bypass uncompressed size limits. If an inbound Cloud JSON-RPC request exceeds the limit before a valid JSON-RPC request object and `id` can be parsed, the daemon must close or reject the request and record an error metric. If a valid request object and `id` can be parsed but the payload exceeds the applicable method limit, the daemon must return JSON-RPC `-32602` (Invalid Params) with `error.data.application_code = 4` (Validation Failed).
*   **REQ-021 (JSON-RPC Error Mapping):** Internal and application errors must map to standard JSON-RPC 2.0 error codes. For application execution errors, the top-level error code must be `-32603` (Internal Error) and the specific application-level code must be returned in the `error.data` object under the `application_code` key:
    *   Standard JSON-RPC wire codes:
        *   `-32700` (Parse Error)
        *   `-32600` (Invalid Request)
        *   `-32601` (Method Not Found)
        *   `-32602` (Invalid Params)
        *   `-32603` (Internal / Busy Error)
    *   Application-specific subcodes carried in `error.data.application_code`:
        *   `1` (Application Error)
        *   `2` (Timeout)
        *   `3` (Local Service Unavailable)
        *   `4` (Validation Failed)
        *   `5` (Rollback Completed)
        *   `6` (Rollback Failed)

*   **REQ-022 (Capability Caching & Lifecycle):** The daemon must populate the capability cache once after the first successful NATS connection and downstream capability responder availability. If initial retrieval fails due to broker/responder unavailability or timeout, the daemon must retry with bounded exponential backoff (e.g., base 2s, max 300s) until the initial cache is populated. After successful initialization, capabilities must not be automatically re-fetched on later NATS reconnect events. The cache must only be refreshed upon detecting a firmware version change, receiving a specific upgrade reboot log, or receiving a valid local management signal.
*   **REQ-023 (TLS v1.3 Security):** All NATS broker connections must enforce TLS v1.3 encryption with strict CA certificate verification configured using local CA paths. Plain text or insecure NATS connections must be rejected.
*   **REQ-024 (Payload Compression):** The client must support WebSocket `permessage-deflate` (RFC 7692) extension negotiation. Unconditional application-level gzip of JSON-RPC text frames is strictly prohibited to maintain OWGW wire compatibility. When `permessage-deflate` is successfully negotiated with the Cloud controller, the underlying WebSocket library must transparently compress outbound frames exceeding `compression_threshold_bytes` (default: 2048 bytes / 2KB) and decompress inbound frames.
*   **REQ-025 (Transaction Retry Policy):** The Request Manager must implement a strict transaction retry policy for idempotent, read-only downstream queries (e.g., `capabilities.get`, `status.get`). Retries must adhere to the following lifecycle:
    1. Maximum of 3 total attempts (1 initial attempt + 2 retries).
    2. A single transaction and `rpc_id` must be retained across all attempts.
    3. Each attempt utilizes an independent NATS request timeout.
    4. The transaction maintains an overall maximum deadline covering all attempts and randomized exponential backoff delays (e.g. base 2s).
    5. The first valid terminal reply wins. Late replies from earlier attempts must be gracefully ignored via the atomic terminal sequence.
    State-changing actions (`configure`, `reboot`, `factory`, `upgrade`, `certupdate`, `reenroll`, `script`) must fail fast on the first error without automatic retries. The `status.get` retry policy applies only to requests sent by the uCentral client to the downstream device/local agent.
*   **REQ-026 (Desired/Applied Cloud Reconciliation Contract):** If a desired configuration write to JetStream KV succeeds but publication of the corresponding `config.apply` trigger fails, the daemon must retain the desired configuration in KV and return a failure to the Cloud. Recovery and reconciliation of desired versus applied configuration state are owned by the Cloud control plane, not by the uCentral client.
*   **REQ-027 (JSON-RPC ID Preservation & Edge Cases):** The daemon must support both string and numeric types for JSON-RPC `id` fields per the JSON-RPC 2.0 specification. The exact Cloud JSON-RPC ID must be preserved in the Request Manager and final WebSocket response. NATS commands and replies must use only the generated internal `rpc_id`. The daemon must enforce the following edge cases:
    *   **Notifications:** Read-only requests without an `id` (notifications) must be executed, but the daemon must not send a JSON-RPC response. State-changing or security-sensitive requests without an `id` must be immediately rejected (dropped), as their critical outcomes cannot be communicated back to the Cloud. To prevent transaction collisions and ensure proper NATS correlation, locking, and timeout tracking, the Request Manager must deterministically construct an `rpc_id` (e.g., session_id:jsonrpc_id) for valid notifications while retaining a flag indicating `respondToCloud=false`. Null or empty strings must never be used as a transaction key.
    *   **Invalid/Malformed IDs:** Requests where `id` is an object, array, or malformed must be rejected immediately with `-32600` (Invalid Request) or `-32700` (Parse Error), and the error response must return `id: null`.
    *   **Reused IDs:** If a valid `id` matches a completed transaction in the Request Manager cache within the same active Cloud session, the daemon must immediately replay the cached response without re-executing the command downstream. A matching ID from a different Cloud session must not use the old cache entry.
*   **REQ-028 (NATS Envelope Serialization Contract):** Internal structures representing NATS commands and triggers must strictly serialize into the documented JSON schema keys (e.g., `version`, `rpc_id`, `command_type`, `action`) without mutation or field loss before transmission.
*   **REQ-029 (Graceful Teardown):** Upon receiving a SIGTERM or SIGINT signal, the daemon must attempt a graceful teardown with a strict bounded deadline of 5 seconds. During this window, Priority-0 transaction responses must receive preferential flushing. Remaining lower-priority messages are discarded if the deadline expires, or earlier if necessary to preserve the deadline for Priority-0 delivery. Upon successful queue depletion or expiration of the deadline, the daemon must force-close all WebSocket and NATS connections and terminate the process with exit code 0.
*   **REQ-030 (Startup Configuration Validation):** The daemon must strictly parse and validate all provided configuration parameters (including device serial, Cloud URL, NATS credentials, queue capacities, and timeout/TTL environment variables) prior to attempting any network connections. Timeout and TTL environment variables (e.g., `OLG_TIMEOUT_DISPATCH`, `OLG_TIMEOUT_CONFIGURE`, `OLG_CACHE_TTL_CONFIGURE`) must be parsed as valid Go durations. Missing variables must safely fall back to documented defaults. Missing mandatory fields, unsupported protocols (e.g., non-WSS or non-TLS URLs), missing TLS certificates, zero/negative queue capacities, or malformed/negative durations must be treated as fatal validation errors. If validation fails, the daemon must log the exact validation failure, avoid starting any background network connection loops, and exit immediately with a non-zero exit status.
*   **OWGW Wire Compatibility Rule:** The Cloud-facing WebSocket method name, JSON-RPC ID, parameter names, JSON types, optionality, response nesting and command-specific status semantics must exactly match the current `ra-wlan-cloud-ucentralgw/PROTOCOL.md`. Internal NATS envelopes may differ, but their translation must neither rename nor omit OWGW fields. OLG-specific fields and methods must be negotiated extensions and must not alter the baseline OWGW exchange.
*   **REQ-031 (OWGW Configure Protocol Compatibility):** The daemon must parse and respond to the Cloud `configure` command according to the OWGW uCentral protocol, including numeric UUIDs, standard and compressed configuration inputs, and the nested configure result/status response structure. The immediate-only `when` handling is an intentional OLG protocol deviation.
*   **REQ-032 (Out of Scope Features):** The `rrm`, `wifiscan`, `fixedconfig`, `powercycle`, `request`, controller-side event buffer retrieval, and `transfer` commands, as well as the `cfgpending` notification lifecycle, are intentionally out of scope for this phase (OLG hardware lacks AP radios). The implementation is wire-compatible for the supported command subset. Unsupported uCentralGW commands must return JSON-RPC -32601 Method Not Found.
*   **REQ-033 (OWGW Reboot Protocol Compatibility):** The daemon must parse and respond to the Cloud `reboot` command according to the OWGW uCentral protocol. This includes mapping internal NATS results to the nested OWGW reboot status error format (0=rebooting, 1=busy, 2=rejected). The immediate-only `when` handling is an intentional OLG protocol deviation/limitation.
*   **REQ-034 (OWGW Factory Protocol Compatibility):** The daemon must parse and respond to the Cloud `factory` command according to the OWGW uCentral protocol, validate the target serial, `keep_redirector`, and scheduling parameters (enforcing the intentional OLG immediate-only protocol deviation), forward the operation through the NATS action contract, and translate the downstream result into the required nested factory status response while preserving the original JSON-RPC ID.
*   **REQ-035 (OWGW Trace Protocol Compatibility):** The daemon must parse the `trace` command (parameters like duration, packets, network, interface, uri), forward it via NATS, and return the trace response.
*   **REQ-036 (OWGW Ping Protocol Compatibility):** The daemon must parse the `ping` command, forward it, and translate the response to the specific `serial`, `uuid`, and `deviceUTCTime` format.
*   **REQ-037 (OWGW LEDs Protocol Compatibility):** The daemon must parse the `leds` command (validating pattern and duration), forward it, and return the specific LEDs status response.
*   **REQ-038 (OWGW Telemetry Protocol Compatibility):** The daemon must parse telemetry stream configurations (`interval`, `types`), forward them, and handle telemetry events asynchronously.
*   **REQ-039 (OWGW Remote Access / RTTY Compatibility):** The daemon must parse the `remote_access` command (validating `params.method == "rtty"`, token, server, port, user), forward it, and return the remote_access status response, preserving the optional `meta` field in the response. The internal NATS action is `rtty`.
*   **REQ-040 (OWGW Certupdate Protocol Compatibility):** The daemon must parse the `certupdate` command, validate the target serial and certificates parameter. Validate base64 syntax and calculate the decoded payload size using a bounded decoder. Reject decoded payloads exceeding 2 MB. The client must not unpack, inspect, persist, or log the certificate archive. Forward it to NATS as `action: "certupdate"`, and map the exact OWGW response. Automatic retries are not permitted. The client MUST NOT write certificate private-key contents to normal logs or audit payloads.
*   **REQ-041 (OWGW Reenroll Protocol Compatibility):** The daemon must parse the `reenroll` command, validate the target serial and `when` constraints, forward it to NATS as `action: "reenroll"`, and return the required reenroll response. This is a state-changing, security-sensitive command that must be serialized alongside `configure`, `reboot`, and `factory`, generating audit logs and managing a strict state lock.
*   **REQ-042 (OWGW Script Protocol Compatibility):** The daemon must parse the `script` command, validate script types (shell, ucode, bundle), ensure `script` is provided with strict base64 validation (and that no non-standard `scriptId` is advertised or required), validate URI payloads and oversized script limits, forward it to NATS, and strictly translate the `error`, `result_64`, `result_sz`, or `result` response format. The daemon must enforce execution timeouts and strictly prohibit logging script secrets or full output to audit streams.
*   **REQ-043 (Strict WebSocket Lifecycle and Security):** The WebSocket Cloud connection must strictly implement mutual isolation and security:
    1.  **Authentication:** The Cloud WSS dial must require strict TLS certificate validation against a configured CA, along with explicit client certificate configuration.
    2.  **Handshake & Permitted Methods:** The daemon must perform a JSON-RPC `connect` exchange upon socket open. While protocol state is `Verifying` or `Rejected`, the daemon may process only WebSocket Ping/Pong control frames and the Cloud JSON-RPC `ping` health method. No other JSON-RPC method is permitted. All other requests must be rejected with `-32603` and `application_code = 3` (Local Service Unavailable).
    3.  **Goroutine Lifecycle:** The reader and writer loops must run as sibling goroutines under an error group. If one fails, the other must be cleanly canceled. Overlapping reconnect sessions must be prevented using connection generation tracking.
    4.  **Heartbeat & Writers:** The writer loop must exclusively send Ping frames and respect write timeouts. The reader loop must handle Pong frames to extend the read deadline.
    5.  **Session Isolation & Priority Policy:** Outbound messages in the scheduler must declare a `SessionID`. The writer loop must enforce the following isolation policy:
        *   **Priority 0 (JSON-RPC Responses):** Strictly session-bound. The writer must discard Priority 0 messages whose `SessionID` does not match the active session. This prevents cross-session leakage where a reused JSON-RPC ID could cause a stale response to be delivered incorrectly.
        *   **Priority 1-3 (State, Health, Audit, Telemetry):** Session-independent. These are device-global events and are safe to replay after reconnect. The writer must process these regardless of a `SessionID` mismatch, ensuring critical asynchronous events are not dropped during connection flaps.
